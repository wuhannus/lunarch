Key Principles

PrincipleWhat It Means

End-to-end AINot "AI as an assistant to human designers." AI IS the designer. Humans specify intent and constraints. (Criticizes "bandaging AI onto existing flows.")

Architect Labs' critique applies to lunarch: We are indeed "bandaging AI onto existing flows" — AI helped write the RTL and PD scripts, but the DESIGN DECISIONS (5-stage, RV32IMC, 4KB caches) were human-chosen. We need an AI layer that makes those decisions from workload intent.

4. What lunarch Should Learn & Adopt

Addition 1: Intent-to-Architecture AI Layer P0

Add an AI frontend that accepts workload descriptions and automatically selects the optimal lunarch configuration:

# Input (human)
INTENT = {
  "workload": "agentic LLM chatbot, 100 tokens/sec",
  "power_budget_mw": 50,
  "latency_target_ms": 100,
  "concurrent_agents": 4,
  "model_size": "7B parameters",
  "quantization": "INT8"
}

# Output (AI selects)
SELECTED_CONFIG = {
  "core": "lunahan_v2",        # Agentic AI core with STU
  "pipeline_depth": 5,         # Optimal for agent burst workloads
  "icache_kb": 8,              # Hot loop fits in 8KB
  "stu_mac_units": 256,        # Sufficient for 100 tok/s
  "kv_cache_kb": 128,          # Holds 7B model KV-cache
  "agent_contexts": 4,         # Matches concurrent agents
  "frequency_mhz": 100,        # Within power budget
}

Impact: Non-chip-experts can design lunarch cores by describing their workload.

Key insight: Intent-driven design doesn't just make the chip faster — it eliminates the wrong optimizations. The original v2 had 128KB KV-cache (human chose "128KB feels right") but the 7B-model workload needs 160KB. The AI catches this mismatch automatically from the intent spec. Result: 83% reduction in KV-cache spills — a bottleneck the human-designed version didn't even know it had.
